## 핵심 주장

"메모리가 있다와 없다 사이의 격차는 종종 서로 다른 LLM 백본 간의 간극보다 더 크다."

에이전트 성능의 진짜 차별화는 모델 선택도, 프롬프트도 아닌 **메모리 아키텍처**다.

---

## 쓰기-관리-읽기 루프

대부분 구현이 Write + Read만 하고 Manage를 무시함:

- **Write**: 새 정보가 메모리에 들어감 (관찰, 결과, 성찰)
- **Manage**: 유지·가지치기·압축·통합 (가장 어렵고 가장 자주 실패)
- **Read**: 관련 기억 검색 → 컨텍스트 주입

---

## 4가지 시간적 메모리 범위

### 1. 작업 기억 (Working Memory)
= 컨텍스트 창. 일시적·대역폭 높음·한계 있음.
실패 모드: 주의력 희석("중간에 묻히는" 현상), 컨텍스트가 길어질수록 성능 저하.

### 2. 일화 기억 (Episodic Memory)
= 구체적 경험 타임라인. "언제 무슨 일이 있었나."
OpenClaw 구현: 일일 스탠드업 로그 → 검색 가능한 타임라인 축적.

### 3. 의미 기억 (Semantic Memory)
= 추상화된 지식·사실·휴리스틱.
OpenClaw 구현: 각 에이전트 MEMORY.md (선별 저장 — 모든 것 넣지 않음).
**선별 단계 없으면 쓰레기 서랍이 됨.**

### 4. 절차적 기억 (Procedural Memory)
= 실행 가능한 기술·행동 패턴.
OpenClaw 구현: AGENTS.md + SOUL.md (페르소나·행동 제약·에스컬레이션 규칙).
가장 소홀히 다뤄지는 영역. 피드백 메커니즘과 반복 갱신이 핵심.

---

## 5가지 메커니즘 계열

1. **컨텍스트 상주 압축**: 슬라이딩 윈도우, 롤링 요약, 계층적 압축
2. **검색 증강 저장소**: 에이전트 이력에 적용된 RAG (임베딩 품질이 병목)
3. **반성적 자기계발**: Reflexion·ExpeL 같은 사후 분석 저장 시스템
4. **계층적 가상 컨텍스트**: MemGPT 스타일 (RAM/디스크/콜드 스토리지)
5. **정책 학습 관리**: 재생 학습으로 최적 호출법 학습 (유망하나 미성숙)

## 주요 실패 모드

### 컨텍스트 실패
- **요약 드리프트**: 반복 압축 → 원본과 어긋난 기억만 남음
- **주의력 희석**: 100만 토큰 창도 중간 정보는 실질적으로 무시됨

### 검색 실패
- **의미론적 불일치**: 유사해 보이지만 관련 없는 기억 반환
- **기억 실인증**: 데이터는 있지만 에이전트가 다시는 꺼내지 못함
- **조용한 조율 실패**: 오류 없이 응답만 점점 나빠짐 (가장 위험)

### 지식 무결성 실패
- **숙성된 상태**: 세계는 변했지만 메모리는 2024년에 고정
- **자기 강화 오류**: 나쁜 기억을 진실로 취급 → 세계관 오염
- **과도한 일반화**: 좁은 맥락의 교훈을 모든 상황에 적용

### 환경 실패
- **모순 처리**: 상충하는 정보 사이를 오락가락 (N8N 워크플로우 사례)

## 설계 긴장 (Trade-offs)

| 축 | 긴장 |
|----|------|
| 효용 vs 효율성 | 좋은 메모리 = 더 많은 토큰·지연·저장 |
| 효용 vs 적응성 | 지금 유용한 기억도 언젠가 식어버림 |
| 적응성 vs 신실성 | 업데이트할수록 원본 왜곡 위험 |
| 충실함 vs 거버넌스 | 정확한 기억에 PHI/PII 포함 가능 |

## 실무 권고 7가지

1. **명시적 시간적 범위부터** — 필요할 때만 각 레이어 구축
2. **관리 단계를 진지하게** — 무엇을 저장·압축·삭제할지 설계
3. **원시 에피소드 기록 유지** — 요약만 믿지 말 것
4. **버전 반사 메모리** — 타임스탬프/버전으로 충돌 판단
5. **절차적 기억을 코드처럼** — 소스 관리 하에 두고 리뷰
6. 파일 기반 메모리는 소규모에 적합, 대규모는 벡터DB·AgentCore 필요
7. 관리 없는 축적은 소음·모순·시스템 저하로 이어짐
