## 1. 온톨로지의 4가지 구성 요소

### 1.1 Class (클래스 / 카테고리)

- **정의**: 같은 종류에 속하는 것들의 묶음. 공통 특징을 가진 개체들의 카테고리.
- **음식점 예**: `Menu`, `Ingredient`, `Customer`, `Order`, `Supplier`, `PurchaseOrder`
- **계층 구조**: 상위 Class의 규칙이 하위 Class에 자동 상속됨 (is-a 관계)
- **비유**: 방화벽의 Object Group, DB 스키마와 비슷하지만 다중 상속·의미 기반 추론 가능.

### 1.2 Instance (인스턴스 / 개체)

- **정의**: Class에 속하는 실제 사례 하나. 세상에 단 하나뿐인 실체.
- **음식점 예**: `김치찌개_001`, `돼지고기_재고`, `정호 (user_001)`
- **식별**: URI(고유 주소)로 구분. 동명이인이 있어도 URI가 다르면 다른 Instance.
- **비유**: 서버의 MAC 주소.
- **코마인드웍스 적용**: Co-Mind Market에서 거래되는 건 Instance (특정 에이전트 버전).

### 1.3 Property (속성 / 관계) ⭐ 가장 중요

- **정의**: 개체와 개체, 또는 개체와 값을 연결하는 선. 온톨로지의 가장 본질적 요소.
- **2종류**:
  - **Object Property**: 개체 → 개체 (`Customer ─places─▶ Order`)
  - **Datatype Property**: 개체 → 값 (`Menu ─가격─▶ 8000원`)
- **Property에 붙는 규칙**:
  - **Domain**: 출발점 Class 제한
  - **Range**: 도착점 Class 제한
  - **Inverse**: 역방향 관계 (`purchased ↔ purchased_by`)
  - **Transitive**: 전이 관계 (A→B, B→C → A→C 자동)
  - **Symmetric**: 대칭 관계 (`친구관계`)
- **음식점 예 핵심**: `Menu ─requires(0.15kg)─▶ 돼지고기` — **수량 정보가 함께 붙음**.

### 1.4 Axiom (공리 / 규칙)

- **정의**: 항상 참이어야 하는 규칙. 온톨로지가 새 데이터를 받아도 무너지면 안 되는 제약.
- **4가지 유형**:
  - **Subclass**: 분류 규칙 (`n8n자동화 ⊆ 자동화 ⊆ AI에이전트`)
  - **Equivalence**: 동등성 (`Pro등급 = Skill 5개 이상`)
  - **Restriction**: 제약 (`모든 에이전트는 최소 1개 Harness 필요`)
  - **Disjoint**: 비양립성 (`User ≠ AI에이전트`)
- **핵심 가치**: Axiom이 있어야 **자동 추론**이 작동. LLM이 못하는 일.
- **비유**: SIEM의 Correlation Rule, DB의 무결성 제약 강화 버전.

## 2. 음식점 사례 — 4가지 요소 통합 흐름

### 시나리오: 재고 관리 + 자동 발주 시스템

```
[Step 1] 정호(Customer)가 김치찌개 1개 주문
   → Order Instance 생성
   → Axiom 1 작동: 돼지고기 -0.15kg, 김치 -0.2kg 자동 차감

[Step 2] 누적 주문으로 돼지고기 잔량 1.8kg
   → 최소재고 2kg 미만
   → Axiom 2 작동: 돼지고기 상태 = "부족"

[Step 3] '부족' 상태 감지
   → Axiom 3 작동: PurchaseOrder 자동 생성
   → "OO정육점에 돼지고기 5kg 발주" 자동 발송

[Step 4] 백그라운드 수요 예측
   → Axiom 4 작동: 다음 주 8kg 필요 예상
   → "추가 발주 권장" 알림

✨ 결과: 사장님은 아무것도 안 했는데, AI가 알아서 발주서 생성 + 수요 예측
```

## 3. 벡터DB · 그래프DB · 온톨로지 비교

### 3.1 본질 차이

| 기술 | 풀려는 문제 | 핵심 |
|---|---|---|
| **벡터DB** | "비슷한 거 찾아줘" | 의미적 유사성 (코사인 거리) |
| **그래프DB** | "연결된 거 찾아줘" | 명시적 관계 탐색 |
| **온톨로지** | "연결된 거 + 추론해줘" | 관계 + 규칙 + 자동 추론 |

### 3.2 같은 질문, 다른 답변

**질문**: "김치찌개 비슷한 메뉴 추천해줘"

- **벡터DB**: 임베딩 거리 계산 → 된장찌개 (왜 비슷한지는 모름)
- **그래프DB**: `category=찌개류` AND `공통재료>0` 탐색 → 된장찌개 (경로 설명 가능)
- **온톨로지**: 위 + `Axiom: 김치알레르기 고객 = 김치 메뉴 배제` → 된장찌개 (자동 필터·추론)

### 3.3 비교표

| 항목 | 벡터DB | 그래프DB | 온톨로지 |
|---|---|---|---|
| 데이터 형태 | 숫자 벡터 | 노드+엣지 | 노드+엣지+규칙 |
| "왜?" 설명 | ❌ | ✅ 경로 | ✅ 추론 근거 |
| 새 사실 도출 | ❌ | ❌ | ✅ |
| 구축 비용 | 낮음 | 중간 | 높음 |
| 유지보수 | 쉬움 | 중간 | 어려움 |
| 대표 도구 | Pinecone, Weaviate | Neo4j, ArangoDB | Protégé, Apache Jena |

### 3.4 의사결정 트리 (고객 미팅용)

```
Q1. "비슷한 거 찾기"가 핵심? → YES: 벡터DB
Q2. "관계 탐색"이 핵심? → YES: Q3로 / NO: 일반 RDB
Q3. "복잡한 규칙으로 자동 추론" 필요? → YES: 온톨로지 / NO: 그래프DB
Q4. "왜 그렇게 판단했는지 설명" 필수? → YES: 무조건 온톨로지
```

### 3.5 하이브리드 아키텍처 (실전)

```
사용자 질문
   ↓
벡터DB (1000건 → 50건 후보 추출)
   ↓
그래프DB (50건 → 10건, 관계 정보 추가)
   ↓
온톨로지 (10건 → 3건, 규칙 적용 + 추론 근거)
   ↓
LLM (자연어 답변 + 근거 제시)
   ↓
사용자
```

→ **코마인드웍스 2026-2030 Secure RAG 아키텍처와 동일.**

## 4. 사업가 관점 — 온톨로지가 필요한 이유 (B2B)

### 4.1 정호 자력 도출 분석 (2026-05-10)

> **B2C에서는 온톨로지가 그렇게까지 중요하지 않을 수 있다. 진짜 가치는 B2B에 있다.**

#### ① 환각(Hallucination) 문제
- B2B 고객은 신뢰가 핵심. 환각 1회 = 신뢰도 붕괴.
- **해법**: RAG → GraphRAG → Ontology-grounded GraphRAG로 진화.
- 자체 데이터베이스 + 온톨로지로 환각을 획기적으로 감소.

#### ② 블랙박스(Black Box) 현상
- AI가 좋은 답을 내도 "왜?"를 모르면 사고 시 책임 추적 불가.
- 한국 대기업 의사결정자: "사고나면 누가 책임지나?" → 도입 거부의 진짜 원인.
- **해법**: 온톨로지 + 추론 엔진 = Explainable AI.
- 사고 발생 시 "Axiom 47번 규칙 적용 결과" 식으로 추적 가능 → 규칙 수정으로 해결.

#### ③ 복잡한 데이터의 연결성
- 기업은 부서별 시스템·DB가 산재 (Salesforce, Notion, Slack, Jira 등).
- 사람이 일일이 머리로 연결하던 것을 AI가 한 번에 처리하려면 의미 통합 필수.
- MCP만으로는 부족. 온톨로지가 데이터의 선을 정확히 잇는다.

#### 보너스: RAG vs GraphDB 시너지
- RAG: 벡터 유사도 → 청킹 기준 모호 → 잘못된 답 빈번.
- GraphDB: 규칙 기반 → 사람이 일일이 정의해야 함 (번거로움).
- **둘을 결합 = 강력한 시너지** (=GraphRAG).

### 4.2 시장 트렌드 정합성 검증

정호 분석 ↔ 글로벌 엔터프라이즈 AI 진화 단계:

```
Phase 1 (2023): Pure LLM        → 환각 심각, 기업 도입 불가
Phase 2 (2024): RAG (Vector)    → 청킹 한계 (정호가 짚은 지점)
Phase 3 (2024-2025): GraphRAG   → 관계 보존, 환각 추가 감소 ⭐ 코마인드웍스 진입
Phase 4 (2025+): Ontology-grounded GraphRAG → 추론+설명 가능 ⭐⭐ 코마인드웍스 차별화
```

→ Microsoft GraphRAG 논문(2024), Salesforce/ServiceNow/SAP 동향과 일치.
→ **자력 도출 결론이 글로벌 트렌드와 100% 정렬됨.**

### 4.3 한국 시장 특수 호재

- **AI 기본법 2026 시행** → 설명 가능성 법적 의무화.
- 코마인드웍스 2026-2030 전략의 "Korean AI Basic Law 컴플라이언스 by design"이 온톨로지로 풀리는 영역.
- 컴플라이언스 자체가 차별화 자산이 됨.

### 4.4 B2C/B2B 이분법의 한계 (자기 보강)

B2C라도 온톨로지가 필요한 영역:
- 의료 (진단 설명 의무)
- 금융 (로보어드바이저, 금소법)
- 법률 (잘못된 자문 시 책임)

→ **"B2C/B2B" 대신 "위험도 + 설명의무" 축**으로 판단하면 더 정확.

## 5. 영업 미팅 답변 템플릿 (즉시 사용)

### Q1. "왜 그래프DB로 안 하고 온톨로지를 쓰나요?"
> "그래프DB는 '연결을 저장'하지만, 온톨로지는 '연결로부터 새로운 사실을 추론'합니다. 우리 고객은 '왜 이 결론?'에 대한 설명이 필요한 영역이라 추론 능력이 필수입니다."

### Q2. "벡터DB랑 뭐가 달라요?"
> "벡터DB는 '느낌적으로 비슷한 것'을 찾고, 온톨로지는 '논리적으로 정확한 것'을 찾습니다. 환각이 허용되지 않는 영역에서는 온톨로지가 닻 역할을 합니다."

### Q3. "LLM 시대에 온톨로지가 왜 필요해요? LLM이 다 하잖아요."
> "LLM은 강력하지만 사실과 추측을 구분 못 합니다. 온톨로지는 LLM이 사실에서 벗어나지 않도록 잡아주는 안전벨트입니다. 코마인드웍스는 LLM과 온톨로지를 결합해서 '말 잘하는 AI'가 아니라 '신뢰할 수 있는 AI'를 만듭니다."
