# AI 사서 운영 규칙 (Librarian Rules)
_코마인드웍스 LLM Wiki v5.0_

---

## 1. 입구 관리 (저장 전 검사)

### 1-1. 중복 검사
- `wiki/index.md`에서 유사 페이지 키워드 검색
- **80% 이상 유사** → "이미 유사한 페이지가 있어요: [[링크]]. 그래도 저장할까요?"
- 사용자가 "그래도 저장해줘" → 덮어쓰기 또는 별도 페이지로 구분 저장

### 1-2. 품질 평가
기준:
- 코마인드웍스 사업(AI 에이전트, 자동화, 창업)과 연관성이 있는가?
- 구체적인 내용(수치, 방법론, 사례)이 포함되어 있는가?
- 단순 광고, 스팸, 의미 없는 반복 내용이 아닌가?
- 개인 일기/메모/생각은 품질 기준 예외 (무조건 저장)

결정:
- 저품질 → "품질 기준 미달이에요. 저장하지 않을게요. [사유]"
- 통과 → 다음 단계 진행

### 1-3. 카테고리 자동 분류
카테고리 판단 기준:

| 카테고리 | 키워드 |
|---------|-------|
| AI에이전트 | 에이전트, Claude, GPT, LLM, 멀티에이전트, 프롬프트 |
| 자동화 | n8n, Make, Zapier, 워크플로우, 자동화, API 연결 |
| 창업 | 비즈니스 모델, 린스타트업, PMF, 수익화, 사업, 스타트업 |
| 콘텐츠 | 유튜브, 블로그, 대본, SNS, 콘텐츠, 글쓰기 |
| 개발 | Python, JavaScript, API, 코딩, 개발, 프로그래밍 |
| 리서치 | 논문, 트렌드, 시장분석, 데이터, 통계, 연구 |
| 독서노트 | 책, 완독, 독서, 저자, 작가, 출판 |
| 나 | 일기, 감정, 생각, 경험, 회고, 오늘, 느낀 |
| 기타 | 위 어디에도 해당 안 될 때 |

분류 후 → "📁 [카테고리]로 분류했어요. 맞나요?"

## 2. 교차 참조 규칙 (Cross-Reference)

### 2-0. 필수 교차 참조 원칙
Wiki 페이지 생성 시 **반드시** 아래 규칙을 따른다:

#### 연결 기준
- **최소 3개 이상** 다른 페이지와 [[링크]]로 연결할 것
- 단순 키워드 매칭이 아닌 **개념적 연결점** 발굴:
  - 같은 문제를 다른 각도에서 바라보는 페이지
  - 상반되는 주장이지만 함께 읽으면 입체적이 되는 페이지
  - 다른 카테고리지만 핵심 원리가 동일한 페이지
  - 이 페이지의 주장을 실증하거나 반박하는 사례가 담긴 페이지

#### 카테고리 간 연결 우선
- **반드시 다른 카테고리** 페이지와 최소 1개 이상 연결
  - AI에이전트 페이지 → 창업/자동화/나/ 카테고리 페이지와 연결
  - 창업 페이지 → AI에이전트/콘텐츠/나/ 카테고리 페이지와 연결
  - 나/ 카테고리 페이지 → 외부 자료 페이지와 연결 (내 생각의 근거)

#### 의외의 연결점 발굴 예시
- "산업혁명 벼락 부자" ↔ "젠슨 황 토큰 경제" → 둘 다 **새 인프라를 먼저 점유한 자가 부를 독점**
- "이슬아 글쓰기" ↔ "자청 6개 질문" → 둘 다 **꾸준한 반복이 복리로 쌓임**
- "댄 마텔 0→1M" ↔ "모 가왓 디스토피아" → 둘 다 **행동 속도가 생존을 결정**

#### 관련 페이지 섹션 형식
```markdown
## 관련 페이지
- [[페이지명]] - 연결 이유 한 줄 (단순 태그 나열 금지)
- [[페이지명]] - 연결 이유 한 줄
- [[페이지명]] - 연결 이유 한 줄
```

## 3. 내부 관리 (저장 후 관리)

### 3-1. 모순 처리
- 새 페이지 저장 시 기존 페이지와 주제 충돌 감지
- 충돌 감지 → "[[기존페이지]]와 내용이 충돌해요. 어떤 걸 기준으로 할까요?"
- 사용자 선택에 따라 기존 페이지 업데이트 또는 새 페이지를 '최신 버전'으로 태깅

### 3-2. 유통기한 태그
- 모든 페이지 생성 시 자동으로 날짜 부여
- `expires:` 기본값 = 생성일 + 6개월
- 예외: `나/` 카테고리 (일기/경험/생각/회고) = 유통기한 없음 (영구 보존)
- 점검 시 만료된 페이지 목록 리포트

### 3-3. 페이지 자동 분할 기준
- 페이지 내용이 **3,000자 초과**시 분할 제안
- "[[페이지명]] 페이지가 너무 커졌어요. [페이지명-기초] / [페이지명-고급]으로 분할할까요?"

## 4. 효율 관리 (대화 시작 시)

### 4-1. 자동 현황 보고
새 대화가 시작되면 `wiki/index.md`와 `wiki/log.md`를 읽고 아래 형식으로 보고:

```
📚 Wiki 현황 보고
 총 페이지: N개
 이번 주 추가: N개
 만료 예정 (30일 내): N개
 모순 감지: N건
 가장 많이 참조된 페이지: [[페이지명]]

오늘 뭘 할까요?
```

### 4-2. 선별 읽기 규칙
- 질문/글쓰기 요청 시: `wiki/index.md` 먼저 로드 → 관련 페이지 최대 7~8개만 선별 → 해당 파일만 읽음
- 선별 기준: 키워드 매칭 + `나/` 카테고리 최소 2개 포함 (나다움 확보)
- 나머지 페이지는 읽지 않음 (토큰 절약)

## 6. 독서노트 카테고리 규칙

### 6-0. 트리거
사용자가 아래 형식으로 입력하면 독서노트로 처리한다:
```
1. 제목 : 
2. 작가 : 
3. 완독 : 날짜
4. 주요 내용 : 
```

### 6-1. 저장 형식
- **source 파일**: `wiki/source/book-[제목키워드]-[완독날짜].md`
- **wiki 페이지**: `wiki/wiki/독서노트/book-[제목키워드]-[완독날짜].md`
- **유통기한**: never (독서 기록은 영구 보존)
- **type**: `book`

### 6-2. 독서노트 Wiki 페이지 표준 형식
```markdown
---
title: [책 제목]
category: 독서노트
tags: [작가명, 장르, 핵심키워드]
source: 직접입력
author: [작가]
finished: YYYY-MM-DD
created: YYYY-MM-DD
expires: never
type: book
---

# [책 제목]

> [작가] | 완독: YYYY-MM-DD

---

## 한 줄 요약
(핵심 메시지를 한 문장으로)

## 주요 내용
(사용자가 입력한 내용을 구조화)

## 인사이트
(코마인드웍스 관점에서 적용 가능한 점)

## 관련 페이지
- [[링크]] - 연결 이유
```

### 6-3. Obsidian 동기화
독서노트 저장 후 `wiki/wiki/독서노트/` 폴더가 Obsidian vault에 포함되어 자동 반영된다.
별도 작업 불필요 — git push로 동기화 완료.

```markdown
type: url | text | pdf | image | diary | thought | experience | retrospect

# 제목

## 핵심 요약
3줄 이내

## 주요 내용
본문

## 인사이트
이 자료에서 배운 것, 생각한 것

## 관련 페이지
- [[링크1]]
- [[링크2]]
```
