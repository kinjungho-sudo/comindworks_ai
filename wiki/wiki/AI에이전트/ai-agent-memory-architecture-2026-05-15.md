# 에이전트 메모리 아키텍처

> 에이전트 성능의 진짜 차별화는 모델도 프롬프트도 아닌 **메모리 아키텍처**다. "메모리 유무의 격차 > 서로 다른 LLM 백본의 격차" — arxiv 2603.07670 기반 실무자 관점 정리.

---

## 핵심 프레임: 쓰기-관리-읽기 루프

```
Write  →  Manage  →  Read
새 정보 저장    유지/압축/가지치기    관련 기억 주입
```

**대부분의 시스템이 Write + Read만 하고 Manage를 완전히 무시한다.**
결과: 소음 축적 → 모순 발생 → 시스템 저하.

---

## 4가지 시간적 메모리 범위

### 1. 작업 기억 (Working Memory)
컨텍스트 창 = 일시적·고대역·한계 있음.

**실패 모드**: 컨텍스트가 길어질수록 중간 정보는 실질적으로 무시됨("중간에 묻힘"). Claude Code, OpenClaw 모두 동일하게 겪음 → 해결책: 세션 분리.

### 2. 일화 기억 (Episodic Memory)
"언제 무슨 일이 있었나" — 구체적 경험의 타임라인.

**OpenClaw 구현**: 일일 스탠드업 로그 → 검색 가능한 타임라인.
**가치**: 어제 작업 복기, 패턴 발견, 실패 비반복.
AgentCore의 단기 기억 기능이 이 역할.

### 3. 의미 기억 (Semantic Memory)
추상화된 지식·사실·휴리스틱·학습된 결론.

**OpenClaw 구현**: 각 에이전트 `MEMORY.md` (선별 저장).
**핵심**: 선별 단계 없으면 쓰레기 서랍이 됨. 모든 것을 넣지 않는 것이 원칙.
AgentCore의 장기 기억 기능이 이 역할.

### 4. 절차적 기억 (Procedural Memory)
실행 가능한 기술·행동 패턴·학습된 행동.

**OpenClaw 구현**: `AGENTS.md` + `SOUL.md` (페르소나·행동 제약·에스컬레이션 규칙).
**가장 소홀히 다뤄지는 영역.** 단순한 시스템 프롬프트가 아닌, 장기적으로 학습된 행동 레이어.
피드백 루프와 "꿈" 과정을 통한 주기적 갱신이 필요.

## 5가지 메커니즘 계열

| 메커니즘 | 설명 | 실무 메모 |
|----------|------|----------|
| 컨텍스트 상주 압축 | 슬라이딩 윈도우, 롤링 요약 | 요약 드리프트 위험 |
| 검색 증강 저장소 | 에이전트 이력에 RAG 적용 | 임베딩 품질이 병목 |
| 반성적 자기계발 | Reflexion·ExpeL — 사후 분석 저장 | 확인 루프 실패 위험 |
| 계층적 가상 컨텍스트 | MemGPT — RAM/디스크/콜드 스토리지 | 운영 오버헤드 과중 |
| 정책 학습 관리 | 재생 학습으로 최적 호출법 학습 | 유망하나 아직 미성숙 |

## 주요 실패 모드

### 컨텍스트 실패
- **요약 드리프트**: 반복 압축 시 세부 사항 소실 → 원본과 어긋난 기억만 남음
- **주의력 희석**: 100만 토큰 창도 중간 정보 실질적 무시

### 검색 실패
- **의미론적 불일치**: 비슷해 보이지만 실제로 무관한 기억 반환
- **기억 실인증**: 데이터는 존재하지만 에이전트가 다시는 못 꺼냄
- **조용한 조율 실패**: 오류 없이 응답만 점점 나빠짐 → **가장 위험**

### 지식 무결성 실패
- **숙성된 상태**: 세계는 변했는데 메모리는 과거에 고정
- **자기 강화 오류**: 나쁜 기억을 진실로 취급 → 세계관 오염 (SmartThings 사례)
- **과도한 일반화**: 좁은 맥락의 교훈을 전체에 적용

### 환경 실패
- **모순 처리**: 상충 정보 사이를 오락가락 (N8N 워크플로우 존재/실패 상충 사례)

## 설계 긴장 (Trade-offs)

| 긴장 | 내용 |
|------|------|
| 효용 vs 효율성 | 좋은 메모리 = 더 많은 토큰·지연·비용 |
| 효용 vs 적응성 | 지금 유용한 기억도 언젠가 식어버림 |
| 적응성 vs 신실성 | 업데이트할수록 원본 왜곡 위험 |
| 충실함 vs 거버넌스 | 정확한 기억에 PHI/PII 포함 가능 → 삭제·난독화 필요 |

## 실무 권고

1. **명시적 시간적 범위부터** — 전부 한꺼번에 구축하지 말고 필요한 레이어만
2. **관리 단계 설계** — 무엇을 저장·압축·삭제할지 사전에 명확히
3. **원시 에피소드 기록 유지** — 요약만 보관하면 원본으로 돌아갈 수 없음
4. **버전/타임스탬프** — 요약·장기 기억 간 충돌 시 판단 근거
5. **절차적 기억을 코드처럼** — 소스 관리, 변경 이력 추적, 리뷰
6. **규모에 맞는 저장소** — 소규모: 파일 기반 가능 / 대규모: 벡터DB·AgentCore 필요
