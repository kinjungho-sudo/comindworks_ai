# Claude Code 토큰 최적화 가이드

> 출처: https://www.youtube.com/watch?v=gLZ1wJUADqk

Claude Code 사용 시 과도한 토큰 소모 문제를 해결하고, 비용 효율적인 개발 워크플로우를 구축하기 위한 전략.

---

## 1. 개요

Claude Code는 강력한 에이전트 기능을 제공하지만, 대화가 길어지거나 대규모 코드베이스를 다룰 때 토큰이 급격히 소모된다. AI가 매 프롬프트마다 **전체 대화 맥락을 처음부터 다시 읽는** 구조적 이유 때문이다.

---

## 2. 토큰 소모의 주요 원인

- **반복적 맥락 읽기**: 새 메시지만 읽는 게 아니라, 매 요청마다 대화 전체를 처음부터 끝까지 다시 읽음
- **불필요한 잔여 데이터**: 종료된 작업의 에러 로그, 테스트 결과 등이 컨텍스트에 남으면 매번 토큰 소모 + 환각 현상 증가

---

## 3. 6가지 핵심 최적화 전략

### 3.1 플랜별 모델 선택 및 고정

- Pro Plan 사용자는 기본 모델이 Opus인지 확인 필요
- Opus는 Sonnet 대비 **4~5배** 토큰 소모
- **액션**: 설정에서 기본 모델을 Sonnet으로 고정, 특수한 경우에만 Opus 호출

### 3.2 모델 자동 전환 (Opus-Plan) 활용

- 설계는 Opus, 실제 코딩은 Sonnet이 수행하도록 자동화
- `/model opus-plan` 명령어로 Plan/Execute 단계에서 모델 자동 전환

### 3.3 컨텍스트 초기화 및 세션 관리

- 기능 구현이나 에러 수정 완료 후 `/clear`로 컨텍스트를 비울 것
- 비우기 전 `/rename`으로 세션 이름 지정 → `/resume`으로 나중에 재접근 가능

### 3.4 확장된 사고 (Extended Thinking) 조절

- Thinking 과정도 출력 토큰으로 과금됨
- 간단한 파일 수정 시 `/config`에서 Thinking 모델을 끄거나 Budget을 낮게 설정

### 3.5 구체적인 프롬프트 작성 (Targeted Prompting)

| 나쁜 예 | 좋은 예 |
|--------|--------|
| "코드 개선해줘" | "src/api/auth.ts의 validateToken 함수를 검증해줘" |
| "버그 찾아줘" | "cart.js 156번째 줄 계산 로직 확인해줘" |

대상을 명확히 지정할수록 AI가 읽는 파일 수가 줄어 토큰 절감.

### 3.6 메모리 파일 및 스킬 분리

- `CLAUDE.md`는 매 세션마다 자동 로드 → **500줄 이하** 유지
- 핵심 규칙만 `CLAUDE.md`에 남기고, 상세 지식이나 가끔 쓰는 로직은 **Skills로 분리**
- 스킬은 필요할 때만 로드되므로 평소엔 토큰 소모 없음

---

## 4. 용어 정리

| 용어 | 설명 |
|------|------|
| **토큰 (Token)** | AI가 텍스트를 처리하는 최소 단위 |
| **컨텍스트 윈도우 (Context Window)** | AI가 한 번에 처리할 수 있는 토큰의 총량 (작업 공간) |
| **컨텍스트 롯 (Context Rot)** | 불필요한 정보가 쌓여 맥락이 오염되고 AI 성능이 저하되는 현상 |

---

## 5. 요약

| 구분 | 권장 사항 |
|------|----------|
| 모델 | Sonnet 고정, Opus는 설계 단계에만 최소 사용 |
| 컨텍스트 | 작업 완료 시 즉시 `/clear`, 필요한 것만 유지 |
| 프롬프트 | 파일명·함수명 명시, 포괄적 요청 금지 |
| CLAUDE.md | 500줄 이하, 나머지는 Skills로 분리 |
